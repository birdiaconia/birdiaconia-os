# Birdiaconia OS Workflow Standard v1.0

## 문서 목적

이 문서는 버디아코니아 OS에서 하나의 작업이 어떻게 시작되고, 누가 무엇을 맡고, 어떤 기준으로 검토되며, 언제 완료되는지를 정하는 **작업 표준**이다.

버디아코니아 OS의 작업은 대화나 아이디어로 바로 완료되지 않는다. 중요한 일은 GitHub Issue로 정리하고, Codex가 작업한 결과는 Pull Request로 제출하며, 사람이 검토한 뒤 Merge되어야 공식 반영된다.

이 표준의 핵심 흐름은 다음과 같다.

```text
Issue 생성
↓
Codex 작업
↓
Pull Request 생성
↓
사람 검토
↓
Merge
↓
완료 기록
```

---

## 1. 기본 원칙

버디아코니아 OS의 모든 작업은 다음 원칙을 따른다.

1. **대화는 입력이다.**
   - 회의, ChatGPT 대화, 현장 메모는 작업의 출발점이다.
   - 그러나 대화 자체는 공식 작업 완료가 아니다.

2. **Issue는 실행 단위다.**
   - 실제로 처리할 일은 GitHub Issue로 만든다.
   - Issue에는 목적, 결과물, 완료 조건이 있어야 한다.

3. **Codex 작업은 브랜치에서 진행한다.**
   - Codex는 main 또는 기본 브랜치에 직접 반영하지 않는다.
   - 작업마다 새 브랜치를 만들고 변경 사항을 분리한다.

4. **Pull Request는 검토 요청이다.**
   - PR은 “완료 선언”이 아니라 “검토해 주세요”라는 요청이다.
   - 운영자 또는 지정 검토자가 내용을 확인해야 한다.

5. **Merge는 공식 반영이다.**
   - Merge된 내용만 버디아코니아 OS의 공식 기준으로 본다.
   - Merge되지 않은 작업은 초안 또는 검토 중 상태다.

6. **사람이 최종 결정한다.**
   - ChatGPT와 Codex는 제안하고 작성하고 정리할 수 있다.
   - 우선순위, 승인, Merge, 외부 공개는 사람이 결정한다.

---

## 2. 역할 구분

버디아코니아 OS 작업에는 사람, ChatGPT, Codex, GitHub가 함께 참여한다. 각자의 역할은 다음과 같이 구분한다.

| 참여자 | 쉬운 설명 | 할 수 있는 일 | 하면 안 되는 일 |
|---|---|---|---|
| 사람 | 최종 책임자와 운영자 | 방향 결정, 우선순위 확정, Issue 승인, PR 검토, Merge 결정 | 검토 없이 AI 결과를 공식 기준으로 확정하지 않기 |
| ChatGPT | 기획·정리 보조자 | 대화 요약, Issue 초안 작성, 완료 조건 정리, 검토 질문 제안 | 최종 승인자처럼 결정하지 않기 |
| Codex | 파일 작업 실행자 | 브랜치 생성, 문서 작성, 코드 수정, 테스트, Commit, PR 작성 | 사람 승인 없이 Merge하거나 운영 결정을 확정하지 않기 |
| GitHub | 작업 기록장과 검토 공간 | Issue, Branch, Commit, PR, Review, Merge 기록 보관 | 내용의 옳고 그름을 스스로 판단하지 않기 |

### 역할을 한 문장으로 구분하기

- **사람**은 “할지 말지와 받아들일지”를 결정한다.
- **ChatGPT**는 “무엇을 해야 하는지”를 정리한다.
- **Codex**는 “파일로 실제 변경”을 만든다.
- **GitHub**는 “모든 과정의 기록”을 남긴다.

## 3. 표준 작업 흐름

버디아코니아 OS의 표준 작업 흐름은 다음 6단계다.

| 단계 | 상태 | 핵심 질문 | 주요 책임 |
|---|---|---|---|
| 1 | Issue | 무엇을 하려는가? | 사람, ChatGPT |
| 2 | Codex 작업 | 결과물을 어떻게 만들 것인가? | Codex |
| 3 | PR | 검토할 수 있게 제출되었는가? | Codex, GitHub |
| 4 | 검토 | 받아들여도 되는가? | 사람 |
| 5 | Merge | 공식 반영할 것인가? | 사람, GitHub |
| 6 | 완료 | 완료 기록과 후속 작업이 정리되었는가? | 사람, ChatGPT |

---

## 4. 1단계: Issue 생성

Issue는 버디아코니아 OS에서 작업을 시작하는 공식 단위다.

### Issue에 반드시 들어갈 내용

| 항목 | 설명 |
|---|---|
| 제목 | 무엇을 하는 작업인지 한눈에 알 수 있어야 한다. |
| 목표 | 왜 이 작업을 하는지 설명한다. |
| 결과물 | 최종으로 만들어져야 할 문서, 파일, 구조를 적는다. |
| 완료 조건 | 어떤 상태가 되면 완료로 볼지 적는다. |
| 주의사항 | 문체, 범위, 제외할 내용, 운영상 유의점을 적는다. |
| 담당 또는 실행자 | 사람, ChatGPT, Codex 중 누가 주로 맡는지 표시한다. |

### 좋은 Issue의 기준

좋은 Issue는 Codex에게 그대로 맡겨도 작업 범위가 흔들리지 않는다.

예시:

```text
목표:
버디아코니아 OS 작업 표준 v1.0을 작성한다.

결과물:
docs/os/Workflow_Standard_v1.md

완료 조건:
- 한국어로 작성한다.
- 사람, ChatGPT, Codex, GitHub의 역할을 구분한다.
- Issue → Codex 작업 → PR → 검토 → Merge → 완료 흐름을 표준화한다.
```

### Issue 단계 체크리스트

- [ ] 작업 목적이 분명한가?
- [ ] 결과물 파일 또는 산출물이 정해졌는가?
- [ ] 완료 조건이 확인 가능한 문장으로 적혀 있는가?
- [ ] 사람이 결정해야 할 부분과 Codex가 작성할 부분이 구분되었는가?
- [ ] 관련 문서나 이전 작업이 연결되어 있는가?

---

## 5. 2단계: Codex 작업

Codex 작업은 Issue를 실제 파일 변경으로 바꾸는 단계다.

### Codex가 작업을 시작할 때 해야 할 일

1. Issue의 목표와 완료 조건을 확인한다.
2. 현재 브랜치 상태를 확인한다.
3. 새 작업 브랜치를 만든다.
4. 관련 문서를 읽고 문체와 구조를 맞춘다.
5. 필요한 파일을 생성하거나 수정한다.
6. 가능한 범위에서 검토 또는 테스트를 실행한다.
7. 변경 내용을 Commit한다.
8. PR을 만든다.

### 브랜치 이름 규칙

브랜치 이름은 어떤 Issue를 위한 작업인지 알 수 있어야 한다.

권장 형식:

```text
issue-{번호}-{짧은-작업명}
```

예시:

```text
issue-13-workflow-standard
issue-14-research-template
issue-15-project-board-fields
```

### Codex 작업 중 지켜야 할 원칙

- Issue 범위를 벗어난 큰 변경은 하지 않는다.
- 다른 사람이 작업 중인 내용을 임의로 되돌리지 않는다.
- 문서 작업은 기존 문서의 표현 방식과 구조를 따른다.
- 운영자가 이해하기 쉬운 표현을 우선한다.
- 변경 이유를 Commit과 PR에 남긴다.

---

## 6. 3단계: Pull Request 생성

Pull Request는 Codex가 만든 작업을 사람이 검토할 수 있도록 제출하는 단계다.

### PR의 의미

PR은 다음 뜻을 가진다.

```text
이 변경 사항을 공식 문서 또는 코드에 반영해도 되는지 검토해 주세요.
```

PR은 완료가 아니라 검토 요청이다.

### PR에 들어갈 내용

| 항목 | 설명 |
|---|---|
| 제목 | 변경 내용을 짧고 명확하게 쓴다. |
| 요약 | 무엇을 만들었거나 수정했는지 적는다. |
| 관련 Issue | 어떤 Issue를 해결하는지 연결한다. |
| 검토 포인트 | 사람이 특히 확인해야 할 부분을 적는다. |
| 테스트 또는 확인 | 실행한 검사, 문서 확인, 제한 사항을 적는다. |
| 후속 작업 | Merge 후 별도 Issue가 필요한 경우 적는다. |

### PR 제목 예시

```text
Issue #13 Workflow Standard v1 문서 추가
```

### PR 본문 예시

```text
## 요약
- 버디아코니아 OS 작업 표준 v1.0 문서를 추가했습니다.
- Issue, Codex 작업, PR, 검토, Merge, 완료 흐름을 표준화했습니다.
- 사람, ChatGPT, Codex, GitHub의 역할을 구분했습니다.

## 관련 Issue
- Closes #13

## 확인
- 문서 파일 생성 확인
- 한국어 작성 확인
```

## 7. 4단계: 사람 검토

사람 검토는 버디아코니아 OS의 품질과 책임을 지키는 단계다.

### 검토자가 확인할 내용

| 검토 항목 | 확인 질문 |
|---|---|
| 목적 적합성 | Issue의 목표를 충족하는가? |
| 완료 조건 | Issue에 적힌 완료 조건을 모두 만족하는가? |
| 운영 언어 | 버디아코니아 운영자가 이해하기 쉬운가? |
| 역할 구분 | 사람, ChatGPT, Codex, GitHub 역할이 혼동되지 않는가? |
| 범위 관리 | Issue 범위를 벗어난 변경이 없는가? |
| 위험 요소 | 법률, 개인정보, 예산, 외부 공개 관련 위험은 없는가? |
| 후속 작업 | Merge 후 추가로 해야 할 일이 있는가? |

### 검토 결과의 종류

| 결과 | 의미 | 다음 행동 |
|---|---|---|
| 승인 | 그대로 반영해도 된다. | Merge한다. |
| 수정 요청 | 일부 보완이 필요하다. | Codex 또는 사람이 수정한다. |
| 보류 | 지금 결정하기 어렵다. | Issue 또는 PR에 사유를 남긴다. |
| 반려 | 방향이 맞지 않아 반영하지 않는다. | PR을 닫고 필요하면 새 Issue를 만든다. |

### 검토 원칙

- 문서의 문장이 멋진지보다 운영에 실제로 도움이 되는지를 본다.
- AI가 만든 문장은 반드시 사람이 책임질 수 있는 표현인지 확인한다.
- 외부 제출, 예산, 법률, 개인정보와 관련된 내용은 더 엄격하게 본다.
- 수정 요청은 가능한 한 구체적으로 남긴다.

---

## 8. 5단계: Merge

Merge는 PR의 변경 사항을 버디아코니아 OS에 공식 반영하는 단계다.

### Merge의 의미

Merge는 다음을 뜻한다.

```text
이 변경 사항을 버디아코니아 OS의 현재 공식 기준으로 받아들인다.
```

따라서 Merge는 단순한 버튼 클릭이 아니라 승인 기록이다.

### Merge 전에 확인할 것

- [ ] Issue 완료 조건을 충족했는가?
- [ ] PR 검토가 끝났는가?
- [ ] 필요한 수정 요청이 반영되었는가?
- [ ] 변경 파일이 의도한 범위 안에 있는가?
- [ ] 후속 Issue가 필요한지 판단했는가?

### Merge 후 해야 할 일

- 관련 Issue를 닫거나 완료 상태로 옮긴다.
- Mission Control 또는 Project 보드에서 카드를 Done으로 이동한다.
- 후속 실행이 필요하면 새 Issue를 만든다.
- 중요한 기준 문서라면 관련 문서에서 링크하거나 참조한다.

---

## 9. 6단계: 완료 기록

작업은 Merge만으로 끝나지 않을 수 있다. Merge 후 운영자가 무엇이 완료되었고 다음에 무엇을 해야 하는지 알 수 있어야 한다.

### 완료 기록에 남길 내용

| 항목 | 설명 |
|---|---|
| 완료된 Issue | 어떤 Issue가 완료되었는지 |
| Merge된 PR | 어떤 PR이 공식 반영되었는지 |
| 반영된 파일 | 어떤 문서 또는 코드가 바뀌었는지 |
| 운영상 의미 | 이 변경이 버디아코니아 OS에 어떤 기준을 추가했는지 |
| 후속 작업 | 추가 Issue, 보드 적용, 현장 실행 등이 필요한지 |

### 완료와 실행 완료의 차이

Merge는 저장소에 공식 반영되었다는 뜻이다. 그러나 실제 현장 실행이 끝났다는 뜻은 아닐 수 있다.

예를 들어 `Workflow_Standard_v1.md`가 Merge되면 “작업 표준 문서 작성”은 완료다. 하지만 이 표준을 모든 운영자가 사용하도록 교육하거나 GitHub Project에 적용하는 일은 별도 Issue로 다룰 수 있다.

## 10. 상태 정의

작업 상태는 다음 기준으로 통일한다.

| 상태 | 쉬운 의미 | GitHub에서 보이는 형태 |
|---|---|---|
| Inbox | 아직 정리 중인 입력 | 메모, 초안 카드 |
| Todo | 하기로 정한 일 | Issue |
| Doing | 작업 중 | 브랜치, 진행 중 Issue |
| Review | 검토 중 | Pull Request |
| Done | 공식 반영 완료 | Merge된 PR, 닫힌 Issue |
| Archive | 보관 또는 보류 | 닫힌 카드, 보관 문서 |

### 상태 흐름

```text
Inbox → Todo → Doing → Review → Done → Archive
```

작업이 막히면 무리하게 다음 단계로 넘기지 않는다. 막힌 이유를 남기고 `Blocked` 또는 `Needs Review`로 표시한다.

## 11. 표준 체크리스트

### 운영자 체크리스트

Issue를 Codex에게 맡기기 전에 확인한다.

- [ ] Issue 목표가 분명한가?
- [ ] 결과물 위치가 정해졌는가?
- [ ] 완료 조건이 적혀 있는가?
- [ ] 한국어, 문체, 독자 수준 등 작성 기준이 있는가?
- [ ] 사람이 결정해야 할 부분이 남아 있지 않은가?

### ChatGPT 체크리스트

Issue를 정리하거나 Codex에게 넘기기 전에 확인한다.

- [ ] 대화 내용을 실행 가능한 작업으로 바꾸었는가?
- [ ] 목표와 결과물을 분리해서 적었는가?
- [ ] 완료 조건을 확인 가능한 문장으로 만들었는가?
- [ ] 운영자가 이해하기 쉬운 표현을 사용했는가?
- [ ] Codex가 파일 작업을 시작할 수 있을 만큼 구체적인가?

### Codex 체크리스트

작업을 완료하고 PR을 만들기 전에 확인한다.

- [ ] 새 브랜치에서 작업했는가?
- [ ] Issue의 결과물 파일을 만들거나 수정했는가?
- [ ] Issue의 완료 조건을 모두 반영했는가?
- [ ] 관련 문서의 문체와 구조를 참고했는가?
- [ ] 변경 내용을 확인했는가?
- [ ] Commit을 만들었는가?
- [ ] PR 제목과 본문에 요약, 관련 Issue, 확인 내용을 적었는가?

### 검토자 체크리스트

Merge 전에 확인한다.

- [ ] Issue 목적과 결과물이 일치하는가?
- [ ] 문서가 운영자가 이해하기 쉬운가?
- [ ] 역할과 책임이 명확한가?
- [ ] 과도한 개발자 용어가 없는가?
- [ ] 후속 작업이 필요한 경우 Issue로 분리할 수 있는가?
- [ ] Merge해도 버디아코니아 OS의 공식 기준으로 문제가 없는가?

## 12. 표준 작업 예시

Issue #13을 예로 들면 표준 흐름은 다음과 같다.

| 단계 | 실제 행동 |
|---|---|
| Issue | `Workflow Standard v1.0 작성` Issue를 만든다. |
| Codex 작업 | Codex가 `issue-13-workflow-standard` 브랜치를 만들고 문서를 작성한다. |
| PR | Codex가 `docs/os/Workflow_Standard_v1.md` 변경 사항을 PR로 제출한다. |
| 검토 | 사람이 문서가 운영 표준으로 적절한지 확인한다. |
| Merge | 승인되면 PR을 Merge한다. |
| 완료 | Issue #13을 닫고 보드에서 Done으로 이동한다. |

이 흐름을 따르면 “누가 무엇을 했는지”, “어떤 기준으로 승인되었는지”, “무엇이 공식 반영되었는지”가 GitHub에 남는다.

## 13. 금지 또는 주의 사항

다음 행동은 버디아코니아 OS 표준 흐름에 맞지 않는다.

- Issue 없이 중요한 작업을 진행하는 것
- Codex가 기본 브랜치에 직접 반영하는 것
- PR 없이 파일 변경을 공식 반영하는 것
- 사람이 검토하지 않은 AI 작성물을 공식 기준으로 사용하는 것
- Merge를 현장 실행 완료로 오해하는 것
- 작업 범위가 다른 여러 Issue를 하나의 PR에 섞는 것
- 법률, 예산, 개인정보, 외부 제출 판단을 AI에게 최종 결정시키는 것

## 14. 문서 산출물 기준

버디아코니아 OS 문서를 작성할 때는 다음 기준을 따른다.

| 기준 | 설명 |
|---|---|
| 한국어 우선 | 운영자가 바로 읽고 사용할 수 있게 한국어로 작성한다. |
| 쉬운 표현 | 개발자 용어보다 운영 언어를 우선한다. |
| 표 사용 | 역할, 단계, 체크리스트는 표로 정리한다. |
| 흐름 명확화 | 시작, 검토, 승인, 완료 기준을 분리한다. |
| 책임 구분 | 사람과 AI의 역할을 혼동하지 않는다. |
| 재사용 가능성 | 다음 Issue와 PR에도 반복 적용할 수 있게 작성한다. |

## 15. Workflow Standard v1.0의 적용 범위

이 표준은 다음 작업에 적용한다.

- 버디아코니아 OS 문서 작성
- Research Object, Policy Object, Data Object 등 OS 객체 문서화
- GitHub Issue 기반 운영 작업
- Codex가 수행하는 파일 변경 작업
- PR 검토와 Merge 흐름
- Mission Control 보드 운영

단, 긴급한 현장 대응이나 외부 기관과의 즉시 소통은 별도 운영 판단이 필요할 수 있다. 그런 경우에도 사후에 Issue 또는 기록으로 남겨야 한다.

## 16. 다음 개선 과제

Workflow Standard v1.0이 반영된 뒤 다음 작업을 별도 Issue로 진행할 수 있다.

1. Issue 템플릿 작성
2. Pull Request 템플릿 작성
3. Mission Control 보드 필드와 상태값 적용
4. Codex 작업 요청 문장 템플릿 작성
5. 문서 검토 체크리스트 분리
6. Merge 후 완료 기록 템플릿 작성

이 문서는 버디아코니아 OS 작업 흐름의 첫 번째 공식 표준이다. 앞으로 실제 운영에서 반복 사용하면서 v1.1, v2.0으로 개선한다.
